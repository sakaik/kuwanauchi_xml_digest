# MOJ-XML 202308/202404データ比較の試み（「点」の変化）
## 概要
最新データとひとつ前に公開されたデータの差分抽出を行いたい場合に、これまではポリゴンに注目して対応付けを試みて失敗していた（ポリゴンに付けられたID: F00000xx(通称F-ID)はデータ作成の都度変更されるので新旧紐付けに使えない）。
面(F-ID)を構成する線(C0000xx形式のC-ID)は更に点(P000xx形式のP-ID)から為るが、このP-IDを指し示す形で定義されている主題属性「筆界点」には、変化しない固有のIDが振られているという情報を得た（Thanks to Wada-san!）。
今回は、202308データと 202404データで、この筆界点のIDを照合キーとして使用しての新旧比較を試みたものである。


## 筆界点情報の例
今回着目するキー「点番名」はXMLファイル中で以下の構造になっている。すなわち、空間属性で、点－線－ポリゴン を定義（それぞれ P-ID, C-ID, F-IDが振られる）する空間属性のブロックが終了し、新たに主題属性のブロックが始まる、その中にひとつひとつの点の属性情報である「筆界点」として「点番名」が定義されている形式となる。
なお、主題属性には、筆界点以外にも、「筆界線」(C-IDへリンク）、「筆」（F-IDへリンク）があるが、このいずれも、点番名のようなコードは保持していない。
```
  :
	</空間属性>
	<主題属性>
		<筆界点>
			<点番名>885522</点番名>
			<形状 idref="P000000001"/>
		</筆界点>
		<筆界点>
			<点番名>885523</点番名>
			<形状 idref="P000000002"/>
		</筆界点>
		<筆界点>
			<点番名>885524</点番名>
			<形状 idref="P000000003"/>
		</筆界点>
		<筆界点>
			<点番名>885525</点番名>
			<形状 idref="P000000004"/>
		</筆界点>
    :
```

## 今回確認に用いるデータ
全国を一遍に実施するには件数が多すぎるだろうと予想されたので、まずは「千葉県」に絞っての確認を試みたのだが、これでも多すぎた。
ということで、今回は更に絞って「我孫子市」データでの比較を行った。我孫子市の自治体コードは 12222 と覚えやすいので、ぜひみんなも活用くださいB-)
地元贔屓で我孫子市を選定したのだが、実は9割以上が任意座標データなのは残念なところである。

## コメント、知見、仮説など
- 表1と表2に登場する point_idは重複しない。つまり今回確認したデータの範囲では、「point_idを保持したまま別のファイルに移動している」ようなものは確認されなかった。
- 今回の表には載せていないが、point_idで202308と202404を結合したとき、同じpoint_idを持つポイントはすべて同じ x,yの座標値を保持していた（各公開バージョンでの変化を追跡するためのキーとして使用可能である）。
- point_id自体は千葉県内だけでも、同じIDが何度も登場する。これは仮説であるが同じ自治体の中では被らないように採番しているのではないか（自治体コード5桁＋point_idでユニークとなる？）。後日調査したい
- 同じpoint_idでも、前回と今回とで必ずしも P-IDは一致しない（結果として大部分一致はしているが、基本的には無頓着に(?)連番を採番しているものと思われる）

## これを筆の差分追跡に活かすには
今回、ポイントデータに紐付く「点番名」を用いることで、登記所備付地図データの公開バージョン間での差分抽出を行えそうであるという知見を得た。
最終目標である変更筆の把握に繋げるには以下の手順が必要である。

- 変更が認められた点の P-ID を使用している線 C-IDリストの作成
- 当該C-IDを使用しているポリゴン F-IDリストの作成
- F-IDに紐付く筆情報の獲得

馬鹿正直にこれらの処理を行うためには、今回（すごく大変だった）ポイント情報のデータベース化と同様のことを、線に対しておよびポリゴンに対して実施する必要がある（線は見たところ必ず2点から構成されているが、ポリゴンは構成する線の数が不定なのでデータ構造的にはこの辺りも課題となる）。

今回の手法による検証を千葉県全域、ひいては全国区で実施するには相当なストレージ容量と、それなりの処理性能が必要となろう。
